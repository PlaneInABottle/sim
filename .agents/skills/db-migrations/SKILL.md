# Database Migrations — Sim Studio

> Authoritative guide for the dual-config Drizzle migration architecture, rebase
> procedures, and production troubleshooting.

## Architecture Overview

This project runs **two migration streams** side by side, each with its own
Drizzle config and output directory. The checked-in configs do **not** declare
separate Drizzle tracking tables, so do not assume table-level isolation unless
you verify it in the target database.

### Upstream Migrations (`packages/db/migrations/`)

| Detail | Value |
|--------|-------|
| **Config** | `packages/db/drizzle.config.ts` |
| **Output dir** | `packages/db/migrations/` |
| **Tracking table** | `drizzle.__drizzle_migrations` (default) |
| **Run command** | `bun run db:migrate:upstream` |
| **Owner** | `sim-origin/main` — **never edit manually** |

These are auto-generated by Drizzle Kit from the repo's Drizzle schema inputs.
Each migration is a numbered `.sql` file accompanied by a snapshot JSON in
`migrations/meta/`. Treat `_journal.json` as the authoritative ordered manifest
for upstream migration application.

### Local / Fork Migrations (`packages/db/local-migrations/`)

| Detail | Value |
|--------|-------|
| **Config** | `packages/db/drizzle-local.config.ts` |
| **Output dir** | `packages/db/local-migrations/` |
| **Tracking table** | No separate table declared in repo config; verify live tracking-table behavior before assuming isolation |
| **Run command** | `bun run db:migrate:local` |
| **Owner** | This fork — additions like `workspace_budget`, `chatwoot_conversation_consumptions` |

Local migrations are fork-specific and **must be preserved on every rebase**.
Upstream never touches `local-migrations/` or `drizzle-local.config.ts`.

### Running Both Together

```bash
# ALWAYS use this for a full migration (chains upstream → local)
bun run db:migrate
```

The `db:migrate` script in `packages/db/package.json` chains both configs:

```
drizzle-kit migrate --config=./drizzle.config.ts && drizzle-kit migrate --config=./drizzle-local.config.ts
```

Because the repo configs do not override Drizzle's default migrations table,
shared-table watermark issues remain possible unless the target database proves
otherwise.

## Rebase Rules for Migrations

On every rebase onto `sim-origin/main`, follow these conflict-resolution rules:

| File / Directory | Resolution |
|------------------|------------|
| `packages/db/migrations/meta/_journal.json` | **Accept theirs** (upstream) entirely |
| `packages/db/migrations/*.sql` | **Accept theirs** — new upstream SQL files |
| `packages/db/migrations/meta/*.json` | **Accept theirs** — snapshot files |
| `packages/db/local-migrations/` | **Keep ours** — upstream never touches this |
| `packages/db/drizzle-local.config.ts` | **Keep ours** |
| `packages/db/package.json` (`db:migrate` script) | **Keep ours** — must chain both configs |

### Post-rebase checklist

1. Verify `_journal.json` has no duplicate `idx` values
2. Confirm `local-migrations/` still contains our custom SQL
3. Run `bun run db:migrate` locally to test both chains
4. If deploying to production after a rebase, check for stale migration rows
   (see [Drizzle Watermark Mechanic](#drizzle-watermark-mechanic-critical))

## Drizzle Watermark Mechanic (Critical)

> **See also:** [references/drizzle-watermark.md](references/drizzle-watermark.md)

### How it works

Drizzle tracks applied migrations in the `drizzle.__drizzle_migrations` table:

| Column | Type | Purpose |
|--------|------|---------|
| `id` | `serial` | Auto-increment PK |
| `hash` | `text` | SHA-256 of the migration SQL content |
| `created_at` | `bigint` | Millisecond timestamp (copied from `folderMillis` / `when` in `_journal.json`) |

**The watermark algorithm:**

```
watermark = SELECT MAX(created_at) FROM drizzle.__drizzle_migrations
```

Drizzle then iterates `_journal.json` entries and **only applies** those where:

```
entry.when > watermark
```

This means: **a single row with a future `created_at` causes ALL subsequent
migrations to be silently skipped.**

### Common failure: stale migration rows from rebases

When a local migration row (e.g., from a prior local-migration or development
experiment) has a `created_at` timestamp newer than legitimate upstream rows,
Drizzle's watermark is set too high and real upstream migrations are skipped.

**Symptoms:**
- `index "X" does not exist` — at a `DROP INDEX` step (the `CREATE INDEX`
  migration was skipped)
- `relation "Y" already exists` — at a `CREATE TABLE` step
- `column "Z" of relation already exists`
- Silent deployment success but missing tables/columns

**Fix:** See [references/production-fix-runbook.md](references/production-fix-runbook.md)

## Trigger Pre-existence Issue

Some upstream migrations use `CREATE TRIGGER` (not `CREATE OR REPLACE TRIGGER`).
If triggers already exist from prior hotfixes or manual intervention, these
migrations fail with:

```
trigger "user_table_rows_insert_trigger" for relation "user_table_rows" already exists
```

**Fix:** Drop the conflicting triggers before applying the migration:

```sql
DROP TRIGGER IF EXISTS user_table_rows_insert_trigger ON user_table_rows;
DROP TRIGGER IF EXISTS user_table_rows_delete_trigger ON user_table_rows;
-- Then re-run the migration
```

## Production Connection

```bash
# Export a production connection string in your shell, then connect
# Use the variable name your deployment environment actually provides.
psql "$DATABASE_URL"
```

`DATABASE_URL` is the repo-standard connection-string variable. If your deployment
platform exposes the production database under a different variable name, export
that value into `DATABASE_URL` (or pass it directly to `psql`) before running the
diagnostics below.

### Quick diagnostic queries

```sql
-- Migration count (should match _journal.json entries length)
SELECT COUNT(*) FROM drizzle.__drizzle_migrations;

-- Current watermark
SELECT MAX(created_at) FROM drizzle.__drizzle_migrations;

-- Last 10 migrations (check for timestamp outliers)
SELECT id, created_at, to_timestamp(created_at / 1000.0) AS human_time
FROM drizzle.__drizzle_migrations
ORDER BY created_at DESC LIMIT 10;

-- Verify critical tables exist
SELECT to_regclass('public.credential_member'),
       to_regclass('public.workspace_budget');
```
