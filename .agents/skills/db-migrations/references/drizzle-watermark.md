# Drizzle Watermark Mechanic — Deep Dive

## The `__drizzle_migrations` Table

When Drizzle Kit runs `migrate`, it creates (if not exists) a schema-qualified
tracking table:

```
drizzle.__drizzle_migrations
```

| Column | Type | Description |
|--------|------|-------------|
| `id` | `serial` | Auto-incrementing primary key |
| `hash` | `text` | SHA-256 hex digest of the migration `.sql` file content |
| `created_at` | `bigint` | Millisecond-precision timestamp copied from the journal |

Each successfully applied migration inserts one row into this table.

---

## The `_journal.json` Manifest

Located at `packages/db/migrations/meta/_journal.json`, this file is the
ordered manifest of all migrations:

```jsonc
{
  "version": "7",
  "dialect": "postgresql",
  "entries": [
    {
      "idx": 0,           // sequential index
      "version": "7",     // journal format version
      "when": 1739697832964,  // ← folderMillis (epoch ms)
      "tag": "0000_careless_black_knight",  // migration filename (without .sql)
      "breakpoints": true
    },
    // ...
  ]
}
```

### The `when` field (a.k.a. `folderMillis`)

- **Type:** Unix epoch in milliseconds
- **Source:** Generated by Drizzle Kit at the moment `drizzle-kit generate` is run
- **Maps to:** `created_at` column in `__drizzle_migrations`
- **Critical role:** This is the value compared against the watermark

---

## The Watermark Algorithm

When `drizzle-kit migrate` runs, it executes this logic:

```
1. watermark = SELECT MAX(created_at) FROM drizzle.__drizzle_migrations
   (returns NULL if table is empty → watermark = 0)

2. For each entry in _journal.json (ordered by idx):
     if entry.when > watermark:
       - Read the .sql file: migrations/<tag>.sql
       - Execute the SQL within a transaction
       - INSERT INTO drizzle.__drizzle_migrations (hash, created_at)
         VALUES (sha256(sql_content), entry.when)
     else:
       - SKIP (already applied)
```

**Key insight:** The comparison is `>` (strict greater-than), not `>=`. A
migration whose `when` exactly equals the watermark is also skipped.

## Why Rebase Creates Stale Rows

### The scenario

1. You have a local fork migration with `when: 1771931359973` (a recent timestamp)
2. This migration was applied to production, inserting a row with
   `created_at = 1771931359973`
3. You rebase onto upstream, which adds new migrations with
   `when: 1771528429740, 1771621587420, ...` — timestamps **older** than your
   local migration's timestamp
4. The watermark is `MAX(created_at) = 1771931359973` (from your local migration)
5. All new upstream migrations have `when < 1771931359973` → **silently skipped**

### The result

The production database is missing tables, columns, indexes, or triggers that
upstream migrations were supposed to create. The next migration that references
those objects (e.g., `DROP INDEX`) fails.

## Diagnostic Queries

### Find the current watermark

```sql
SELECT MAX(created_at) AS watermark,
       to_timestamp(MAX(created_at) / 1000.0) AS watermark_human
FROM drizzle.__drizzle_migrations;
```

### List all migration rows (check for outliers)

```sql
SELECT id,
       hash,
       created_at,
       to_timestamp(created_at / 1000.0) AS human_time
FROM drizzle.__drizzle_migrations
ORDER BY created_at DESC
LIMIT 10;
```

### Count total entries (should match journal length)

```sql
-- DB count
SELECT COUNT(*) AS db_count FROM drizzle.__drizzle_migrations;

-- Compare with journal length (run locally)
-- cat packages/db/migrations/meta/_journal.json | jq '.entries | length'
```

### Find stale rows (timestamps beyond last legitimate upstream entry)

To identify stale rows, you need the `when` value of the **last upstream journal
entry**. Check `_journal.json`:

```bash
# Get the last upstream entry's timestamp
cat packages/db/migrations/meta/_journal.json | jq '.entries[-1].when'
```

Then query:

```sql
-- Replace <last_upstream_when> with the value from above
SELECT id, created_at, to_timestamp(created_at / 1000.0) AS human_time
FROM drizzle.__drizzle_migrations
WHERE created_at > <last_upstream_when>
ORDER BY created_at DESC;
```

Any rows returned are stale (from local/fork migrations or development
experiments) and are inflating the watermark.

### Identify which migrations were skipped

```sql
-- Get all applied timestamps
SELECT created_at FROM drizzle.__drizzle_migrations ORDER BY created_at;
```

Compare this list against all `when` values in `_journal.json`. Any journal
entry whose `when` is not present in the DB **and** is less than the current
watermark was skipped.

Script approach:

```bash
# Extract all journal timestamps
jq -r '.entries[].when' packages/db/migrations/meta/_journal.json > /tmp/journal_timestamps.txt

# Query DB for applied timestamps
psql "$PROD_DATABASE_URL" -t -A -c \
  "SELECT created_at FROM drizzle.__drizzle_migrations ORDER BY created_at" \
  > /tmp/db_timestamps.txt

# Find skipped (in journal but not in DB)
comm -23 <(sort /tmp/journal_timestamps.txt) <(sort /tmp/db_timestamps.txt)
```

## How SHA Hashes Are Computed

Drizzle uses **SHA-256** of the migration SQL file content to produce the `hash`
value stored in `__drizzle_migrations`.

```bash
# Compute the hash for a migration file (matches what Drizzle stores)
sha256sum packages/db/migrations/0150_some_migration.sql | cut -d' ' -f1
```

When manually inserting migration tracking rows (after applying SQL by hand),
you must compute and use this exact hash:

```sql
-- Example: manually recording a migration
INSERT INTO drizzle.__drizzle_migrations (hash, created_at)
VALUES (
  '<sha256_hex_of_sql_file>',   -- sha256sum of the .sql file
  <folderMillis_from_journal>   -- the "when" value from _journal.json
);
```

## Prevention

To avoid watermark issues in the future:

1. **Do not assume local migrations are isolated by tracking table** — the
   checked-in `drizzle-local.config.ts` changes the output directory, but it
   does not declare a different Drizzle migrations table. Verify the target
   database before relying on table-level isolation.

2. **On rebase:** Always accept upstream's `_journal.json` entirely (theirs).
   Never merge or interleave local entries into the upstream journal.

3. **Post-rebase deploy:** If any local migration rows were previously written
   to `drizzle.__drizzle_migrations` (from before the dual-config split), they
   must be cleaned up. See the
   [Production Fix Runbook](production-fix-runbook.md).
